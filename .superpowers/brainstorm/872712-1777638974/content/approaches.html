Three UI/UX Directions for code-review-loop

Each is a distinct paradigm — pick the one that fits how you want to work

A

Enhanced CLI + Profile System

Pure terminal, zero dependencies beyond Python

  • Named TOML profiles (~/.config/crl/profiles.toml)
  • crl config edit opens $EDITOR; crl run --profile fast
  • Rich-powered live progress bar + phase tracker
  • Pipeline steps defined declaratively in TOML
  • Sync via dotfiles git repo — dead simple

Pros

  • No new runtime deps
  • Agents use same CLI
  • Fast to build

Cons

  • Config editing is text
  • No visual pipeline
  • Limited "wow" factor
B

TUI Dashboard (Textual)

Interactive terminal app — stays in the shell

  • Full-screen TUI: profile picker, pipeline builder, live log pane
  • Keyboard-driven config forms (no mouse required)
  • Pipeline steps as draggable/reorderable rows
  • Live phase progress with timing + status colours
  • Configs stored as TOML, synced via git/dotfiles

Pros

  • Premium terminal feel
  • Visual without a browser
  • Fast navigation

Cons

  • Textual dependency
  • Pipeline "wiring" is limited vs. GUI
  • SSH sessions need care
C

Local Web Companion

Browser UI launched on demand; CLI stays primary

  • crl ui starts a local server, opens browser
  • Visual node-based pipeline editor (React Flow)
  • Profile manager: create, clone, diff, import/export
  • Real-time run monitor via SSE (no polling)
  • Config JSON synced via cloud (e.g. Syncthing, iCloud Drive, git)

Pros

  • Most "premium" feel
  • Best pipeline editor
  • Rich run history UI

Cons

  • Heavier build (Node/React)
  • Two processes to manage
  • Overkill if rarely used

My recommendation: B (TUI) — gives you the premium feel and visual pipeline editor while staying entirely in the terminal. Agents continue to use the raw CLI unchanged. Option C is the right call if you want a genuine node-graph pipeline editor; Option A if you want zero new dependencies.
